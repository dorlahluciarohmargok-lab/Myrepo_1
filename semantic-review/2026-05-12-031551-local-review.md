# Per-todo timers, card UI with background images, and statistics page

These two commits embed timer state directly into each todo item (mode, duration, elapsed time, background index), add a statistics summary endpoint, and rewrite the frontend todo list from expandable text items into visually-rich cards with inline play/pause controls. The backend gains four new columns on `todos`, a `PATCH /{id}/timer` endpoint for persisting elapsed time, and a `GET /timer/stats/summary` endpoint that computes year-scale aggregation on every call. The frontend drives its own per-todo timer loop via `setInterval`, persisting to localStorage every 5 seconds and only syncing elapsed time to the server on explicit user actions.

**Watch for:** the stats summary endpoint loads all timer records unbounded into memory to find the longest session (confirmed); per-todo timer elapsed time drifts between client and server because the frontend only syncs on create/toggle, not on tick (confirmed); `PATCH /timer` accepts arbitrary integers with no validation, allowing negative elapsed values (confirmed).

## High-level view

The Todo entity now carries its own timer configuration — mode (countdown/stopwatch), configured duration, elapsed seconds, and a cosmetic background-image index. This makes each card a self-contained focus session, but it also means timer state lives in two places: localStorage on the client (updated every 5 seconds) and the database on the server (updated only on explicit sync). These will diverge across sessions, tabs, or devices.

The `getStatsSummary` endpoint pulls up to a year of `FocusStatistic` rows, then separately loads *every* `TimerRecord` for the user to find the longest session. For active users this is an unbounded query that grows linearly with usage and will eventually cause latency spikes or OOM under load.

The frontend rewrites todos as image cards with embedded countdown/stopwatch controls. Only one per-todo timer can run at a time (enforced client-side), but the global page-level timer (countdown/stopwatch/clock tabs) has no relationship to the per-todo timers — both can run simultaneously and both record sessions independently, potentially double-counting focus time.

The new `PATCH /{id}/timer` endpoint accepts a raw integer from an untyped `Map<String, Integer>` body with no bounds checking. A malicious or buggy client can set elapsed to negative values or values exceeding the configured duration, corrupting statistics downstream.

4. **Dual timer systems double-count focus time** — The page-level timer and per-todo timers both call `recordSession()`, feeding the same `stats.sessions` array. A user running both simultaneously records twice the actual focus time. Either disable the global timer when a per-todo timer runs, or deduplicate in `recordSession`.

## Unbounded queries in `getStatsSummary`

The summary endpoint performs two database calls:

```java
List<FocusStatistic> allStats = statisticRepository.findByUserIdAndStatDateBetweenOrderByStatDateAsc(
    userId, LocalDate.now().minusYears(1), LocalDate.now());

List<TimerRecord> records = timerRepository.findByUserIdOrderByStartedAtDesc(userId);
```

The first call is bounded to 365 days (at most 365 rows). The second call loads *every timer record the user has ever created* with no date filter, no pagination. Its only purpose is to compute `max(actualDuration)`. For a user who logs 10 sessions/day over a year, that's 3,650 rows loaded into heap just to find a single integer. This should be a `SELECT MAX(actual_duration) FROM timer_records WHERE user_id = ?` query at the repository level.

The streak calculation then performs a linear search through `allStats` for each day in a 365-day window:

```java
for (int i = 0; i < 365; i++) {
    LocalDate d = checkDate.minusDays(i);
    boolean hasActivity = allStats.stream().anyMatch(s -> s.getStatDate().equals(d));
    ...
}
```

This is O(365 * n) where n is the size of `allStats`. Converting to `Set<LocalDate> activeDates = allStats.stream().map(FocusStatistic::getStatDate).collect(Collectors.toSet())` before the loop reduces it to O(365).

## Timer state split between client and server

The per-todo timer ticks entirely on the client. `startTodoTimer` runs a `setInterval` that increments `todo.timerElapsed` and calls `saveTodos()` (localStorage) every 5 seconds. No network call happens during ticking. The `PATCH /todos/{id}/timer` endpoint exists but is never called by the timer loop — it appears to be designed for manual sync but nothing in the frontend currently invokes it during active timing.

This means:
- If the user closes the tab mid-session, localStorage has the latest elapsed but the server doesn't.
- If the user opens a second tab or device, they see stale `timerElapsed` from whatever was last synced.
- `syncTodosToServer` merges by title (not ID), and doesn't merge `timerElapsed` from the server response back into local state — server todos only get added if they don't exist locally.

The merge logic in `syncTodosToServer` uses `localMap[t.title]` as the key, meaning two todos with the same title will collide silently.

## No validation on timer PATCH endpoint

The controller extracts `elapsed` from an untyped map:

```java
Integer elapsed = body.get("timerElapsed");
```

The service then accepts whatever comes in:

```java
todo.setTimerElapsed(elapsed != null ? elapsed : 0);
```

There's no check that `elapsed >= 0`, no check that it doesn't exceed `timerDuration` for countdown mode, and no `@Valid` annotation on the `@RequestBody`. The TodoCreateRequest DTO similarly has no `@Min`/`@Max` on `timerDuration`. A client sending `{"timerElapsed": -5000}` would be persisted.

## Dual timer architecture with no mutual exclusion

The page has two independent timer systems:

1. **Global timer** — the countdown/stopwatch/clock section at the top, driven by `startCountdown()` / `startStopwatch()`. On completion or pause, it calls `recordSession(duration, mode)` and `saveTimerToServer(...)`.

2. **Per-todo timers** — each card's play button starts `startTodoTimer(id)`, which also calls `recordSession(...)` on countdown completion or stopwatch pause.

Both feed `stats.sessions` (local) and potentially the server's timer records. Nothing prevents running the global countdown while a per-todo timer is also running. A user who does this would see inflated statistics — two sessions recorded for the same real-world time period.

## Index on `timer_mode` is unlikely to be useful

The schema adds `INDEX idx_timer_mode (timer_mode)` on a column with only two possible values (`countdown`, `stopwatch`). This index has extremely low cardinality and will rarely be selected by the query optimizer over a full table scan. It adds write overhead on every INSERT/UPDATE without providing query benefit.
